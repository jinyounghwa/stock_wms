# 재고 기능 설명 및 산식 정리

## 1. 문서 목적

이 문서는 **현재 프로젝트에 구현된 데모 로직 기준**으로 아래 4개 메뉴를 설명합니다.

- 재고조사
- 재고조정
- 위치별 재고현황
- 부진재고

추가로, 프로젝트에 있는 입출고/이동 이력 화면 흐름을 참고해 **향후 입출고 목록이 어떻게 구성되면 좋은지 제안안**도 함께 정리합니다.

> 기준 소스
>
> - 화면/집계 로직: `src/NewWmsApp.jsx`, `src/App.jsx`
> - 기능 정의 문서: `재고2/new/재고조사.md`, `재고2/new/위치별 재고 현황.md`, `재고2/new/부진재고.md`
>
> 주의: 현재 프로젝트는 **실서비스 연동 전 단계의 데모 앱** 성격이 있으므로, 일부 값은 목업 데이터(`PRODUCT_MASTER`, `LOCATION_STOCK_ROWS`, `SLOW_MOVING_SKUS`, `INITIAL_SURVEY_ENTRIES`)를 기준으로 계산됩니다.

## 2. 공통 데이터 기준

### 2-1. 숫자 변환

프로젝트 전반에서 숫자형 입력은 문자열이어도 `toNumber()`를 통해 숫자로 정규화해서 사용합니다.

- 산식: `toNumber(value)`
- 의미: `,` 제거 후 `Number()` 변환, 실패 시 `0`

즉, 문서에서 아래와 같은 산식이 있을 때 입력값이 문자열이어도 실제 계산은 숫자 변환 후 수행됩니다.

### 2-2. 공통 키

재고조사/대조/조정 흐름에서 가장 중요한 식별 키는 아래입니다.

- 상품 식별: `barcode`
- 위치 식별: `locationCode`
- 비교 단위 키: `barcode + locationCode`

즉, 같은 바코드라도 위치가 다르면 **다른 재고 행**으로 취급합니다.

### 2-3. 상태값 해석

현재 프로젝트에서 자주 쓰는 상태는 아래와 같습니다.

- 조사 상태: `조사중`, `실입고완료`, `취소됨`
- 조정 상태: `대기중`, `승인됨`, `거절됨`
- 대조 결과: `일치`, `불일치+`, `불일치-`, `미처리`

## 3. 재고조사

재고조사는 현재 프로젝트에서 크게 아래 흐름으로 구성됩니다.

1. 상품/위치 기준 조사 수량 입력
2. 조사 데이터 누적
3. 전산 현재고와 대조
4. 차이 발생 시 재고조정으로 연계

핵심 구현 기준은 `src/NewWmsApp.jsx`의 `SurveyInputTab`, `CurrentCompareTab` 입니다.

### 3-1. 전산 현재고 생성 방식

전산 기준 재고는 `PRODUCT_MASTER`의 위치별 수량을 펼쳐서 `SYSTEM_STOCK_ROWS`로 만듭니다.

- 원천: `PRODUCT_MASTER[].locations[].qty`
- 생성 단위: 상품 바코드 + 위치코드 1행

산식은 사실상 아래와 같습니다.

- `전산현재고(systemQty) = PRODUCT_MASTER 내 해당 barcode/locationCode의 qty`

예시:

- 바코드 `880123456789`
- 위치 `WHO1-A01-01-01`
- 전산 수량 `150`

이면 현재고 대조 시 기본 현재고는 `150`입니다.

### 3-2. 조사 수량 집계 방식

현재고 대조 화면에서는 조사 입력 데이터를 그대로 1건씩 보지 않고, 같은 상품/위치 기준으로 합산합니다.

대상 데이터:

- `surveyEntries`
- 단, `status !== "취소됨"` 인 건만 포함

집계 키:

- `key = barcode + "|" + locationCode`

산식:

- `조사수량(surveyQty) = Σ entry.qty`
- 조건: 같은 `barcode`, 같은 `locationCode`, 그리고 `취소됨` 제외

즉,

- 같은 위치에서 같은 바코드를 3번 조사했다면 합산됩니다.
- 취소된 조사 데이터는 집계에서 제외됩니다.

### 3-3. 현재고 대조 차이 수량

현재고 대조의 핵심 값은 차이 수량입니다.

산식:

- `차이수량(diff) = 조사수량(surveyQty) - 전산현재고(systemQty)`

해석:

- `diff = 0` → 전산과 실사가 일치
- `diff > 0` → 조사수량이 더 많음
- `diff < 0` → 조사수량이 더 적음

예시:

- 조사수량 `145`
- 전산현재고 `150`
- 차이수량 `145 - 150 = -5`

### 3-4. 대조 결과 판정식

현재 프로젝트의 대조 결과는 아래 규칙으로 결정됩니다.

1. 조사 데이터 자체가 없으면 `미처리`
2. 조사 데이터가 있고 `diff === 0`이면 `일치`
3. 조사 데이터가 있고 `diff > 0`이면 `불일치+`
4. 조사 데이터가 있고 `diff < 0`이면 `불일치-`

판정식:

- `survey 없음` → `미처리`
- `survey 있음 && diff === 0` → `일치`
- `survey 있음 && diff > 0` → `불일치+`
- `survey 있음 && diff < 0` → `불일치-`

### 3-5. 전산에는 없고 조사에는 있는 경우

현재 구현은 `surveyMap`에는 있는데 `SYSTEM_STOCK_ROWS`에는 없는 키도 별도 행으로 추가합니다.

이때 산식은 아래와 같습니다.

- `systemQty = 0`
- `diff = surveyQty - 0 = surveyQty`
- `result = 불일치+`

즉, **실사에서는 잡혔지만 전산에 없는 재고**는 자동으로 `불일치+` 로 취급됩니다.

### 3-6. 현재고 대조 요약 카드 산식

현재고 대조 상단 요약은 필터 적용 이후 `filteredRows` 기준으로 계산됩니다.

#### 1) 총 대조 건수

- `총대조건수 = filteredRows.length`

#### 2) 일치 건수

- `일치건수 = count(result === "일치")`

#### 3) 불일치(+) 건수

- `불일치플러스건수 = count(result === "불일치+")`

#### 4) 불일치(-) 건수

- `불일치마이너스건수 = count(result === "불일치-")`

#### 5) 미처리 건수

- `미처리건수 = count(result === "미처리")`

#### 6) 재고 정확도

- `재고정확도(%) = (일치건수 / 전체건수) × 100`
- 전체건수가 0이면 `0.0`

프로젝트 구현상 표현식:

- `accuracy = total ? ((match / total) * 100).toFixed(1) : "0.0"`

### 3-7. 최소 차이 필터 산식

현재고 대조 화면의 최소 차이 수량 필터는 절대값 기준입니다.

- 적용식: `abs(diff) >= 최소차이`

예시:

- 최소 차이 `5`
- `diff = -3` → 제외
- `diff = +7` → 포함
- `diff = -5` → 포함

### 3-8. 재고조정 연계 생성식

현재고 대조에서 선택된 불일치 항목을 재고조정으로 넘길 때 생성되는 조정 요청 값은 아래와 같습니다.

- `systemQty = 현재고 대조의 전산현재고`
- `actualQty = 현재고 대조의 조사수량`
- `reason = "현재고 대조 불일치"`
- `status = "대기중"`

즉, 대조 화면에서 생성되는 조정 요청의 핵심 수식은 다음과 같습니다.

- `조정수량 = actualQty - systemQty = 조사수량 - 전산현재고`

### 3-9. 재고조사 문서 관점에서 정리해야 할 값

`재고2/new/재고조사.md` 기준으로 재고조사 메뉴에서 설명해야 할 산술값은 아래처럼 정리하는 것이 적절합니다.

- 조사 수량 = 같은 상품/위치에 대해 입력된 조사수량 합계
- 현재고 = 전산 위치재고
- 차이 수량 = 조사 수량 - 현재고
- 재고 정확도 = 일치 건수 / 전체 건수 × 100
- 실산 완료율 = 실산 완료 건수 / 전체 대상 건수 × 100
- 미처리 건수 = 조사 미입력 또는 실입고 미완료 건수

## 4. 재고조정

재고조정은 현재 프로젝트에서 아래 두 축으로 나뉩니다.

- 작업자 화면: 조정 요청 생성/재요청
- 관리자 화면: 승인/반려 처리

핵심 구현 기준은 `src/NewWmsApp.jsx`의 `WorkerAdjustmentPanel`, `AdminAdjustmentPanel` 입니다.

### 4-1. 전산 수량(systemQty) 결정 방식

재고조정 요청 화면에서는 선택된 위치와 상품 기준으로 전산 수량을 구합니다.

#### 1) 위치와 상품이 정확히 매칭되는 경우

- `systemQty = 해당 locationCode 의 재고 qty`

#### 2) 위치 상품 원본에 `stock`, `available`, `reserved` 만 있는 경우

프로젝트는 아래의 최대값으로 안전하게 전산 수량을 구성합니다.

- `systemQty = max(stock, available + reserved, 1)`

이 식은 데모 데이터의 불완전성을 보완하기 위한 로직입니다.

### 4-2. 조정 수량 산식

재고조정의 핵심 수식은 아래와 같습니다.

- `조정수량(adjustmentQty) = 실제수량(actualQty) - 전산수량(systemQty)`

해석:

- `+` 이면 재고 증가 조정
- `-` 이면 재고 감소 조정
- `0` 이면 조정 요청 불가

예시:

- 전산 `150`
- 실제 `155`
- 조정수량 `+5`

### 4-3. 요청 가능 조건

프로젝트에서 재고조정 요청 버튼이 활성화되는 조건은 아래와 같습니다.

- 위치 선택 완료
- 상품 선택 완료
- 실제 수량 입력 완료
- `adjustmentQty !== 0`

즉 논리식은 아래입니다.

- `canSubmitRequest = selectedProduct && effectiveLocation && actualQty !== "" && (actualQty - systemQty) !== 0`

### 4-4. 조정 요청 생성값

작업자가 요청을 생성하면 아래 값으로 저장됩니다.

- `requestedAt = 현재시각`
- `owner = 선택한 화주사`
- `worker = "김작업"`
- `systemQty = 선택 상품의 전산 수량`
- `actualQty = 입력 실제 수량`
- `reason = 입력 사유`
- `status = "대기중"`
- `approvedAt = ""`

여기서 실제 계산 값은 아래 하나가 핵심입니다.

- `조정수량 = actualQty - systemQty`

### 4-5. 조정내역 재요청 산식

반려 또는 미승인 요청을 재요청할 때도 계산식은 동일합니다.

- `재요청 조정수량 = 재입력 실제수량 - 기존 전산수량`

재요청 가능 조건:

- 실제 수량 > 0
- `실제 수량 - 전산 수량 != 0`
- 사유 입력 필수

### 4-6. 관리자 승인/반려 처리

관리자 화면에서 승인/반려 시 현재 프로젝트는 **상태값만 변경**합니다.

#### 승인

- `status = "승인됨"`
- `approvedAt = 현재시각`

#### 반려

- `status = "거절됨"`
- `approvedAt = 현재시각`
- `reason = 반려 사유 또는 기존 사유`

### 4-7. 현재 프로젝트 기준 한계

현재 데모에서는 승인 후에도 원천 재고 데이터(`PRODUCT_MASTER`, `SYSTEM_STOCK_ROWS`, `LOCATION_STOCK_ROWS`)를 실시간으로 다시 계산해 반영하지는 않습니다.

즉, 문서에는 아래처럼 명시하는 것이 정확합니다.

- 현재 구현 기준 승인/반려는 **조정 요청 이력 상태 관리 중심**
- 실서비스 전환 시에는 승인 완료 후 **전산 재고 원장 반영 + 입출고 원장 기록 생성 + 위치별 재고 재계산**이 후속으로 필요

## 5. 위치별 재고현황

위치별 재고현황은 위치 단위로 재고, 가용, 예약, 적재율을 보는 화면입니다.

핵심 구현 기준은 `src/NewWmsApp.jsx`의 `LocationStockPage` 입니다.

### 5-1. 행 단위 기본값

위치별 재고 데이터의 기본 행은 `LOCATION_STOCK_ROWS` 입니다.

각 행은 최소 아래 값을 가집니다.

- `stock`: 위치 총 재고
- `available`: 가용 재고
- `reserved`: 예약(미발송) 재고
- `capacity`: 위치 적재 가능량

### 5-2. 적재 비율(util) 산식

각 위치의 적재 비율은 아래와 같이 계산됩니다.

- `적재비율(util, %) = (stock / capacity) × 100`
- 단, `capacity <= 0` 이면 `0`

예시:

- `stock = 90`, `capacity = 120`
- `util = 90 / 120 × 100 = 75%`

### 5-3. 선택/조회 결과 요약 산식

화면 상단 KPI와 선택 요약은 공통 함수 `summarizeRows(rows)` 기준으로 계산됩니다.

#### 1) 총 재고

- `총재고 = Σ stock`

#### 2) 총 가용 수량

- `총가용 = Σ available`

#### 3) 총 예약 수량

- `총예약 = Σ reserved`

#### 4) 위치 수

- `위치수 = rows.length`

#### 5) 평균 적재 비율

- `평균적재비율 = Σ util / 위치수`
- 위치 수가 0이면 `0`

#### 6) 예약 비율

- `예약비율(%) = (총예약 / 총재고) × 100`
- 총재고가 0이면 `0`

예시:

- 총재고 `1,000`
- 총예약 `120`
- 예약비율 `120 / 1000 × 100 = 12%`

### 5-4. 센터별 요약 산식

센터별 카드에서도 같은 구조를 사용합니다.

- `센터 총재고 = 해당 센터 row.stock 합`
- `센터 총가용 = 해당 센터 row.available 합`
- `센터 총예약 = 해당 센터 row.reserved 합`
- `센터 평균적재비율 = 해당 센터 row.util 평균`
- `센터 예약비율 = 센터 총예약 / 센터 총재고 × 100`

### 5-5. 선택 요약과 전체 요약의 차이

현재 화면은 사용자가 위치를 선택하면 선택 행 기준으로 별도 KPI를 보여줍니다.

- 선택 행이 있으면 `selectionSummary = summarizeRows(selectedRows)`
- 선택 행이 없으면 `selectionSummary = summarizeRows(filteredRows)`

즉, 같은 카드여도 기준 모수가 달라질 수 있습니다.

### 5-6. 위치 밀도/강도 표시 기준

카드 색상과 강도는 `getLocationIntensity(stock, capacity)` 기준입니다.

- `capacity <= 0` → `empty`
- `ratio <= 0` → `empty`
- `ratio <= 30` → `low`
- `ratio <= 60` → `medium`
- `ratio <= 90` → `high`
- 그 외 → `full`

여기서

- `ratio = (stock / capacity) × 100`

### 5-7. 문서에 반드시 적어야 할 위치별 재고 산식

위치별 재고현황 문서에는 아래 산식을 명확히 적는 것이 좋습니다.

- 총재고 = 가용 + 예약 + 기타 상태 재고의 합
- 가용재고 = 즉시 출고 가능한 재고
- 예약재고 = 이미 주문/작업에 할당된 재고
- 예약비율 = 예약재고 / 총재고 × 100
- 적재비율 = 현재 적재수량 / 위치 적재가능수량 × 100
- 평균 적재비율 = 조회 위치들의 적재비율 평균

> 참고: 현재 NewWmsApp 화면에서는 요약값 계산에 `stock`, `available`, `reserved`가 직접 사용됩니다. 실서비스에서는 `stock = available + reserved + hold + damaged ...` 같은 구조를 표준화하면 더 좋습니다.

## 6. 부진재고

부진재고는 위치 단위로 풀어 보여주지만, 필터의 출발점은 SKU 단위입니다.

핵심 구현 기준은 `src/NewWmsApp.jsx`의 `SlowMovingBasePage` 입니다.

### 6-1. SKU 총 재고 산식

부진재고 필터는 각 SKU의 전체 위치 재고를 먼저 합산합니다.

- `SKU총재고 = Σ (availableQty + reservedQty)`

즉, 한 SKU가 여러 위치에 흩어져 있으면 위치별 수량을 전부 합쳐 1개 SKU의 총량으로 봅니다.

### 6-2. 필터 조건 산식

현재 프로젝트에서 부진재고 대상 여부는 아래 조건을 모두 만족해야 합니다.

- 화주사 일치 또는 `ALL`
- 센터 일치 또는 `ALL`
- 시즌 일치 또는 `ALL`
- 브랜드 일치 또는 `ALL`
- `daysNoOutbound >= 기준 무출고일수`
- `daysSinceInbound >= 기준 경과일수`
- `SKU총재고 >= 최소 재고`
- 검색어 매칭

즉, 수치 기준 핵심은 아래입니다.

- `무출고 충족 = daysNoOutbound >= minDaysNoOutbound`
- `입고후 경과 충족 = daysSinceInbound >= minDaysSinceInbound`
- `최소재고 충족 = skuTotalStock >= minStock`

### 6-3. 위치별 수량 및 비중 산식

SKU가 필터를 통과하면, 실제 리스트는 위치 단위로 펼쳐집니다.

#### 1) 위치 수량

- `위치수량(qty) = availableQty + reservedQty`

#### 2) 위치 점유 비중

- `위치비중(locationRatio, %) = 위치수량 / SKU총재고 × 100`
- SKU총재고가 0이면 `0`

이 값은 한 SKU의 물량이 특정 위치에 얼마나 몰려 있는지 보는 지표입니다.

### 6-4. 부진 등급 판정식

현재 프로젝트의 등급은 `daysNoOutbound` 기준입니다.

- `365일 이상` → `위험`
- `180일 이상 365일 미만` → `경고`
- `90일 이상 180일 미만` → `주의`
- `90일 미만` → `정상`

즉,

- `grade = f(daysNoOutbound)`

### 6-5. 상단 요약 카드 산식

부진재고 상단 기본 요약은 아래처럼 계산됩니다.

#### 1) SKU 수

- `SKU수 = unique(filteredRows.skuId).count`

#### 2) 가용 합계

- `총가용 = Σ availableQty`

#### 3) 예약 합계

- `총예약 = Σ reservedQty`

### 6-6. 리스크 우선순위 점수 산식

이 프로젝트에서 부진재고의 핵심 차별점은 `priorityScore` 입니다.

#### 1) 기초 값

- `qty = availableQty + reservedQty`
- `ageWeight = min(1, daysNoOutbound / 365)`
- `zoneWeight = zonePriority[zoneBase]`
- `qtyWeight = qty / maxQty`
- `availableShare = availableQty / qty`

여기서 `zoneWeight` 기본값은 아래입니다.

- 출고존 = `1`
- 입고존 = `0.88`
- 보관존 = `0.46`
- 반품존 = `0.34`
- 뮬랑 = `0.26`
- 기타 = `0.4`

#### 2) 최종 우선순위 점수

- `priorityScore = round((ageWeight × 0.42 + zoneWeight × 0.28 + qtyWeight × 0.20 + availableShare × 0.10) × 100)`

이 식은 아래 의미를 가집니다.

- 오래 안 나간 재고일수록 점수 증가
- 출고존/입고존처럼 핵심 위치를 점유할수록 점수 증가
- 수량이 많을수록 점수 증가
- 가용 비중이 높을수록 실제 처분 가능성이 높아 점수 증가

### 6-7. 조치 수준 판정식

`priorityScore` 에 따라 조치 수준을 나눕니다.

- `75 이상` → `즉시 이동`
- `60 이상 75 미만` → `우선 이동`
- `45 이상 60 미만` → `모니터링`
- `45 미만` → `관찰`

### 6-8. 의사결정 요약 산식

프로젝트는 리스크 행(`riskRows`)을 기반으로 별도 요약을 계산합니다.

#### 1) 조치 대상 위치 수

- `moveLocationCount = count(priorityScore >= 60)`

#### 2) 고위험 위치 수

- `highRiskLocationCount = count(priorityScore >= 70)`

#### 3) 핵심 존 적치 수량

- `primeQty = Σ qty`
- 조건: `zoneBase ∈ {출고존, 입고존}`

#### 4) 고위험 수량

- `highRiskQty = Σ qty`
- 조건: `priorityScore >= 70`

#### 5) 평균 리스크

- `avgRisk = Σ priorityScore / riskRows.length`
- 데이터 없으면 `0`

### 6-9. 존별 요약 산식

존별 요약(`zoneDecisionSummary`)은 아래처럼 계산됩니다.

#### 1) 존 총 수량

- `zone.totalQty = Σ row.qty`

#### 2) 존 가용 수량

- `zone.availableQty = Σ row.availableQty`

#### 3) 존 예약 수량

- `zone.reservedQty = Σ row.reservedQty`

#### 4) 우선 이동 수량

- `zone.priorityQty = Σ row.qty`
- 조건: `priorityScore >= 60`

#### 5) 일반 수량

- `zone.normalQty = zone.totalQty - zone.priorityQty`
- 음수 방지를 위해 `max(..., 0)` 처리

#### 6) 위치 수

- `zone.locationCount = count(rows in zone)`

#### 7) 이동 필요 위치 수

- `zone.moveLocationCount = count(priorityScore >= 60)`

#### 8) 즉시 조치 위치 수

- `zone.urgentLocationCount = count(priorityScore >= 75)`

#### 9) 존 평균 리스크

- `zone.avgRisk = zone.riskTotal / zone.locationCount`
- 위치 수가 0이면 `0`

#### 10) 존 이동 비중

- `zone.moveShare(%) = zone.moveLocationCount / zone.locationCount × 100`
- 위치 수가 0이면 `0`

### 6-10. 시각화 값 산식

부진재고 차트도 각각 산식이 명확합니다.

#### 1) 등급 차트

- `정상 개수 = count(grade === 정상)`
- `주의 개수 = count(grade === 주의)`
- `경고 개수 = count(grade === 경고)`
- `위험 개수 = count(grade === 위험)`

#### 2) 경과일 버킷

- `0~89일`: `count(0 <= daysNoOutbound <= 89)`
- `90~179일`: `count(90 <= daysNoOutbound <= 179)`
- `180~364일`: `count(180 <= daysNoOutbound <= 364)`
- `365일+`: `count(daysNoOutbound >= 365)`

#### 3) 존 히트맵

- `zone.count = 해당 존 위치 행 수`
- `zone.qty = 해당 존의 qty 합`

## 7. 4개 메뉴를 하나의 업무 흐름으로 연결해서 설명하는 방법

실무 문서에서는 4개 메뉴를 개별 설명만 하지 말고 아래처럼 연결해서 설명하는 것이 좋습니다.

### 7-1. 재고조사

- 실물 수량을 입력한다.
- 전산 현재고와 차이를 확인한다.

### 7-2. 재고조정

- 차이가 있는 항목에 대해 조정 요청을 올린다.
- 승인되면 전산 재고를 보정해야 한다.

### 7-3. 위치별 재고현황

- 현재 위치별 적재 상태와 가용/예약 현황을 본다.
- 조정 승인 후 가장 직접적으로 영향받는 화면이다.

### 7-4. 부진재고

- 오랫동안 움직이지 않는 재고를 찾아낸다.
- 위치 효율과 장기 적치 관점에서 재배치/할인/폐기 판단에 활용한다.

즉, 권장 설명 순서는 아래입니다.

- `재고조사 → 재고조정 → 위치별 재고현황 → 부진재고`

## 8. 입출고 목록 제안안

현재 프로젝트에는 `src/App.jsx`에 입출고/이동 데모 화면이 일부 존재합니다. 이를 기준으로 보면 향후 입출고 목록은 **단순 조회 화면**이 아니라, 아래 4개 메뉴와 연결되는 **재고 원장형 화면**으로 가는 것이 가장 좋습니다.

### 8-1. 권장 목적

입출고 목록은 아래 역할을 동시에 해야 합니다.

### 8-2. 권장 데이터 모델

입출고 목록의 1행은 아래 기준을 권장합니다.

- `거래일시`
- `거래번호(txId)`

- `상세유형`
- `화주사`
- `센터`
- `존`
- `로케이션`
- `품목코드`
- `바코드`
- `품목명`
- `옵션`
- `로트번호`
- `유통기한`
- `변동수량`
- `이전수량`
- `변경후수량`

### 8-3. 거래유형 표준 제안

입출고 목록에서 `거래유형` 은 아래처럼 표준화하는 것이 좋습니다.

이렇게 두면 재고조사/재고조정에서 발생한 수량 변동도 일반 입출고와 같은 원장 안에서 관리할 수 있습니다.

### 8-4. 입출고 목록 상단 KPI 제안

#### 1) 총 거래 건수

- `총거래건수 = rows.length`

#### 2) 총 입고 수량

- `총입고수량 = Σ 변동수량`
- 조건: 거래유형이 `입고`, `반품입고`, `재고조정증가`, `실사반영증가`

#### 3) 총 출고 수량

- `총출고수량 = Σ abs(변동수량)`
- 조건: 거래유형이 `출고`, `폐기출고`, `재고조정감소`, `실사반영감소`

#### 4) 총 이동 수량

- `총이동수량 = Σ 이동수량`
- 조건: 거래유형이 `이동`

#### 5) 순변동 수량

- `순변동 = 총입고수량 - 총출고수량`

> 이동은 창고 전체 기준으로는 순변동 0 이므로 KPI에서는 별도 분리하는 것이 좋습니다.

### 8-5. 화면 필터 제안

입출고 목록 필터는 아래 구성이 실무적으로 가장 유용합니다.

- 기간(필수)
- 화주사
- 센터
- 존
- 로케이션
- 거래유형
- 상세유형
- 품목코드/바코드/품목명 검색
- 로트번호
- 작업자
- 참조문서번호
- 원인화면

### 8-6. 4개 메뉴와 입출고 목록 연결 규칙 제안

#### 1) 재고조사와 연결

- 실입고 완료 또는 실사 반영 완료 시 입출고 원장 생성
- 권장 유형:
  - `실사반영증가`
  - `실사반영감소`

산식:

- `변동수량 = 조사수량 - 전산현재고`
- `변경후수량 = 이전수량 + 변동수량`

#### 2) 재고조정과 연결

- 관리자 승인 시 입출고 원장 자동 생성
- 권장 유형:
  - `재고조정증가`
  - `재고조정감소`

산식:

- `변동수량 = 실제수량 - 전산수량`
- `변경후수량 = 이전수량 + 변동수량`

#### 3) 위치별 재고현황과 연결

- 위치 이동 발생 시 원장 2행 또는 1행-쌍 구조 생성
- 권장 방식:
  - 출발 위치 1행 `이동출`
  - 도착 위치 1행 `이동입`

산식:

- 출발 위치 변경후수량 = 출발 이전수량 - 이동수량
- 도착 위치 변경후수량 = 도착 이전수량 + 이동수량

#### 4) 부진재고와 연결

- 부진재고 화면에서 직접 수량을 바꾸기보다는 “조치 실행”으로 연결
- 실행 결과가 아래 거래유형으로 남아야 함
  - 재배치 → `이동`
  - 할인/프로모션 출고 → `출고`
  - 폐기 → `폐기출고`

### 8-7. 권장 목록 정렬 방식

기본 정렬은 아래를 권장합니다.

- 1순위: 거래일시 내림차순
- 2순위: 거래번호 내림차순
- 3순위: 라인번호 오름차순

이렇게 해야 최근 변경 이력을 빠르게 추적할 수 있습니다.

### 8-8. 권장 상세 팝업/드릴다운

입출고 목록은 행 클릭 시 아래를 보여주는 것이 좋습니다.

- 거래 헤더 정보
- 상품 정보
- 위치 변경 전/후 수량
- 관련 문서(실사 차수, 조정 요청 번호, 출고 번호, 입고 번호)
- 승인 이력
- 메모/사유

### 8-9. 최종 제안

현재 프로젝트 기준으로 가장 좋은 방향은 다음입니다.

- 입출고 목록을 단순 “입고/출고 조회”가 아니라 **재고 변동 원장**으로 설계
- 재고조사/재고조정/위치이동/부진재고 후속 처리까지 모두 여기로 귀결
- 모든 수량 변화는 `이전수량`, `변동수량`, `변경후수량` 3개 값으로 설명 가능하게 설계

핵심 표준 산식은 아래 3개입니다.

- `변동수량 = 업무 이벤트로 인해 실제 바뀐 수량`
- `변경후수량 = 이전수량 + 변동수량`
- `순변동 = 총입고 - 총출고`

## 9. 문서화 시 권장 표현

실제 사내 설명문서에는 아래 문장을 그대로 활용해도 좋습니다.

### 재고조사

- 재고조사는 상품/위치 기준으로 입력된 실사 수량을 누적하고, 전산 현재고와 비교하여 차이 수량을 계산하는 기능이다.
- 차이 수량은 `조사수량 - 전산현재고` 로 계산한다.
- 재고 정확도는 `일치 건수 / 전체 건수 × 100` 으로 계산한다.

### 재고조정

- 재고조정은 실물 수량과 전산 수량의 차이를 승인 절차를 통해 보정하는 기능이다.
- 조정 수량은 `실제수량 - 전산수량` 으로 계산한다.

### 위치별 재고현황

- 위치별 재고현황은 로케이션 단위 재고, 가용, 예약, 적재율을 조회하는 기능이다.
- 적재 비율은 `현재 적재수량 / 적재가능수량 × 100` 으로 계산한다.
- 예약 비율은 `예약수량 / 총재고 × 100` 으로 계산한다.

### 부진재고

- 부진재고는 장기간 출고되지 않은 재고를 식별하고 위치 점유 우선순위를 계산하는 기능이다.
- 부진 우선순위 점수는 무출고일수, 존 중요도, 수량 규모, 가용 비중을 가중 합산하여 계산한다.

## 10. 결론

현재 프로젝트 기준으로 보면 4개 메뉴는 각각 분리된 조회 화면이 아니라, 아래 하나의 재고관리 체계로 이해하는 것이 맞습니다.

- 재고조사: 실물 확인
- 재고조정: 차이 승인 처리
- 위치별 재고현황: 현재 상태 조회
- 부진재고: 장기 적치 분석
- 입출고 목록: 모든 재고 변화의 원장

따라서 향후 고도화 우선순위는 아래를 추천합니다.

1. 재고조정 승인 후 실제 재고 원장 반영
2. 입출고 목록을 재고 변동 원장으로 통합
3. 위치별 재고현황과 부진재고가 같은 원장을 바라보도록 정합성 확보
4. 재고조사 차수/실산/실입고 이력을 입출고 원장과 연결
